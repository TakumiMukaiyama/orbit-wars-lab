# Orbit Wars 戦略設計

- 作成日: 2026-04-25
- コンペ締切: 2026-06-23 (残約8週)
- 対象: Kaggle Orbit Wars コンペ提出エージェント

## 1. ゴールと勝利条件の理解

- 勝利条件: ゲーム終了 (最大500ターン) 時点での合計艦船数 (惑星駐留 + フリート) が最多のプレイヤー。
- 早期終了: 1人 (または0人) だけが惑星とフリートを持つ状態になったとき。本設計では早期終了を主目的にしない。駆逐を狙うより、艦船合計値の最大化を最優先とする。
- 2P と 4P の両方に同一エージェントで対応する。4P では隣接2人との近距離戦と、対角1人との太陽越え遠距離戦が非対称に発生する。2位以内を維持しつつ1位を目指す行動が期待値で優位になる。

## 2. 戦略の方針

### 2.1 差別化の軸

ヒューリスティックを積むだけでは Kaggle 上位入賞 (上位数%) には届かない想定。Orbit Wars は完全情報・決定論寄りの挙動 (直線フリート、周期軌道、既知彗星経路) のため、フォワードシミュレーション + 貪欲探索による "次手の先読み" が構造的優位になる。これを差別化の本丸に据える。

段階開発の方針:

- Phase 1: 強いヒューリスティック単独で提出し、LB 上の座標を把握する。Phase 2 の評価基盤にもなる。
- Phase 2: 軽量シミュレータ + greedy rollout でヒューリスティックを上書きする。

### 2.2 Phase 1 に入れる構成要素

1. 目標価値計算と艦船予算: nearest ではなく `value = production × max(0, rival_eta - my_eta) - ships_needed` でターゲットを選ぶ。`rival_eta` はそのターゲット惑星に対する "自分以外の最速到達プレイヤーの ETA" (敵がまだ送っていない場合は ∞)。送る艦船数は最小必要量 + マージン (例 +20%) に抑え、残りを防御に残す。
2. 太陽回避ルーティング: 直線経路が太陽を通る場合、接線を取って迂回。経路延長分だけ ETA を再計算してターゲット評価に戻す。
3. フリート迎撃: 敵フリートの到達先と時刻を予測し、防衛不可能な場合に最小艦で迎撃フリートを出す。同時着弾 (coincident attack) も同じ機構で記述する。

彗星先取りは Phase 1 から外し、Phase 2 の評価関数チューニング時に再検討する。

### 2.3 Phase 2 の探索方式

Greedy rollout を採用する。

- 1 手 = 1 惑星からの 1 アクション。
- 現状態から候補手を列挙 → 各手を K ターン先まで簡易シム → スコア最大の 1 手を採用 → 状態更新 → 次の惑星へ。
- 所有惑星の数だけ繰り返し、合成した複数アクションをエージェント出力とする。

想定計算量: 自惑星10 × 候補20 × rollout深度 `D=15` ≒ 3000 シミュレーションステップ/ターン。1秒予算で収まるかは Phase 1 中に実測する (`scripts/bench_sim.py`)。実測結果次第で beam 化 (Phase 2c) へ拡張する余地を残す。

MCTS/minimax は Python では本プロジェクトでは選ばない。シミュレータ速度が想定を大幅に超える場合のみ再検討する。

### 2.4 行動空間の抽象化

- 各自惑星について、敵および中立の惑星を候補ターゲットとする。
- 送る艦船量は "最小必要量", "最小 + マージン (例 +20%)", "全駐留の半分" の 2-3 段階に離散化する。
- "待機" も候補手に含める。
- ターゲットは距離上位 `N_target` 件 (例 `N_target=8`) に絞り、候補数を一定に保つ。

## 3. モジュール構造

### 3.1 src/

| ファイル | 責務 | 依存 |
|----------|------|------|
| `agent.py` | Kaggle エントリ。obs を受け取り action を返す | targeting, intercept, search |
| `utils.py` | Planet/Fleet/距離/ETA/軌道予測 (既存) | - |
| `geometry.py` | 太陽回避ルーティング、接線計算、象限判定 | utils |
| `targeting.py` | ターゲット価値計算、艦船予算、候補手列挙 | utils, geometry |
| `intercept.py` | 敵フリート予測、迎撃計画、同時着弾 | utils, geometry |
| `simulator.py` | Phase 2: 軽量フォワードシミュレータ (numpy) | utils |
| `evaluator.py` | Phase 2: 状態スコアリング関数 | utils |
| `search.py` | Phase 2: greedy rollout 本体 | simulator, evaluator, targeting |

各モジュールは単一責務。`agent.py` は組み立て役で、Phase 1 では `targeting` + `intercept` + `geometry` を直接呼ぶ。Phase 2 では `search.select_moves(obs)` を呼ぶ形に差し替える。

### 3.2 baselines/

- `v0_nearest.py`: 現行 `src/agent.py` をコピーしてタグ保存。
- `v1_phase1.py`: Phase 1 完了時点の固定。
- 以降バージョンごとに追加。

### 3.3 scripts/

| ファイル | 用途 |
|----------|------|
| `test_local.py` (既存) | vs random のスモークテスト |
| `bench_sim.py` | シミュレータ速度ベンチ (1 ターン何 μs か) |
| `evaluate.py` | vs 過去版 / vs pool の自動評価 (50 / 200 試合の二段) |
| `fetch_public_bots.py` | Kaggle Notebook から公開 bot をダウンロードし `baselines/pool/` に配置 |

### 3.4 tests/

`pytest` による関数単位の回帰テスト:

- `test_geometry.py`: 太陽回避ルートの到達保証、接線計算。
- `test_targeting.py`: value/艦船予算の境界条件。
- `test_intercept.py`: 直線予測、同時着弾タイミング。
- `test_simulator.py`: 少数手番の結果が公式環境の `env.run` と一致する。

## 4. 評価手段

### 4.1 採用するレイヤー

| レイヤー | 用途 | 頻度 |
|----------|------|------|
| vs 過去版 (勝率追跡) | 自己改善の確認、改悪の検出 | 毎改修 |
| vs 固定相手プール (Kaggle 公開 Notebook から収集) | 過学習防止 | 改修単位 |
| Kaggle LB | 最終真実、ローカルとの乖離監視 | 週 2-3 回 |
| pytest | 関数レベル回帰 | 毎 push |

`test_local.py` の vs random はスモークテストとして残す。重要判断には使わない。4P ミックス対戦は時間コストが高いため Phase 2 後半で導入する。

### 4.2 統計基準と運用

- 勝率 ±5% 信頼区間 95% に必要な試合数は約 400。毎改修で重いため二段評価とする。
  - 50 試合スクリーニング: 前バージョンに対し勝率 55% 以上を通過条件とする。
  - 200 試合本試験: 通過した候補のみ回す。
- 試合時間は未計測の仮説値 (1試合 30-60秒)。Phase 1 早期に `bench_sim.py` で実測する。
- Kaggle 公開 Notebook から bot を定期的に取り込み、プールを 3-5 種類に維持する。

## 6. 前提と未計測の数値

CLAUDE.md の原則に従い、以下は "仮置きであり、実装中に計測する" ものとして明記する:

- シミュレータ速度 (1 ターン何 μs): 未計測。W1-W3 で `bench_sim.py` によって実測。
- 1 試合の実時間 (30-60 秒想定): 未計測。Phase 1 評価基盤の立ち上げ時に測定。
- Greedy rollout の actTimeout 1s 内での実行可能深度 `D`: 未計測。W3 の simulator 完成後に実測し、`D` を逆算する。

これらの数値が想定と大きく乖離した場合、Phase 2 の探索方式 (greedy/beam/MCTS) を再選択する余地を残す。

## 7. やらないこと (スコープ外)

- 強化学習 (DQN, PPO 等): 学習時間とシミュレータ速度が見合わない。完全情報 + 決定論的挙動なので探索で足りる想定。
- 独自 GUI ビジュアライザ: Kaggle Environments の `env.render()` で十分。
- 多言語実装 (Rust, C++ 拡張): Python + numpy で actTimeout を超える場合のみ再検討。
- 早期終了 (駆逐) 狙いの積極戦略: 勝利条件は合計艦船最大化なので、駆逐戦略は評価関数の一要素として間接的に表現するに留める。

## 8. Future Work (Phase 2 後半以降で検討)

### 8.1 探索方式の代替案

採用した Greedy rollout 以外に以下の方式を検討したが、Phase 2a 時点では採用しない。Phase 2c で余裕があれば実験する。

| 記号 | 方式 | 特徴 | 実装コスト |
|------|------|------|-----------|
| A | Greedy rollout (採用) | 1 手ずつベスト追加 | 小 |
| B | Beam search | 幅 B で候補保持、協調手が出やすい | 中 |
| C | MCTS / minimax | 敵応手も読む、Python では厳しい | 大 |
| D | Portfolio (スクリプト選択) | 複数ヒューリスティックをシム評価 → 最良採用 | 小 |
| E | Flat Monte Carlo | 候補手ごとにランダム rollout 平均 | 中 |
| F | RHEA (進化的行動列) | 次 H ターンの行動列を進化 | 大 |
| G | RMHC (変異山登り) | 初期解を変異で改善、anytime | 中 |
| H | Iterative Deepening Greedy | Greedy の深度を段階拡張 | 小 |

特に D (Portfolio) と G (RMHC) は Phase 2c で greedy に上乗せできる余地がある。D は Phase 1 ヒューリスティックをそのまま "スクリプト" として流用可能。G は anytime 特性で時間予算の余りを改善に回せる。

### 8.2 劣勢時の目的関数切替 (variance-seeking の正しい位置づけ)

決定論ヒューリスティックは "期待艦船数最大化" を目的関数にするが、劣勢時 (残ターン < 150 かつ艦船比 < 0.2 など) には期待艦船数最大化が "負けを受け入れる" 挙動となり勝率が 0 に張り付く。Kaggle のレーティングシステム (TrueSkill 想定) は期待勝率で評価されるため、勝率 0 の戦略より "期待艦船数は下がるが勝率は上がる" 戦略に切り替える必要がある。

注: これは "分散を増やすと Kaggle で有利" という主張ではない。レーティングは期待勝率で決まるので、分散自体に価値はない。ただし劣勢時の最適戦略が平常時と異なるだけ。

実装候補:
- Boltzmann ターゲット選択: `P(target) ∝ exp(value / τ)`。τ を不利度連動にする。
- Portfolio の劣勢時重み付け: 高リスクスクリプト (オールイン等) の採用確率を上げる。
- 評価関数の劣勢モード: evaluator に "敵ホーム占領時の勝率ボーナス" を加算する条件分岐。

Phase 2b で評価関数チューニング時に実験し、有効なら組み込む。事前仮説に基づく実装ではなく、劣勢局面のリプレイを分析して必要性を判断する。
